# 타 시스템 연동 설계

이 문서는 게시판 프로젝트에서 수행하는 **타 시스템 연동**을 정의한다.

본 프로젝트의 연동 대상은
**Google News RSS(IT 뉴스) 피드**이며,
서버는 외부 데이터를 수집하여 **API 응답 형태로 제공**한다.

구현 코드는 포함하지 않는다.

---

## 1. 연동 목적

* 클라이언트는 외부 데이터(IT 뉴스)를 우리 서버 API만 호출하여 뉴스 목록을 반환받는다
* 외부 시스템 장애가 우리 서버 전체 장애로 확장되지 않도록 한다

---

## 2. 연동 대상(External System)

### 2-1. 데이터 소스

* Google News RSS
* 주제: IT
* 피드 URL은 구성값으로 관리한다

---

## 3. 연동 방식

본 프로젝트의 연동 방식은 다음과 같다.

* 서버가 외부 RSS를 호출하여 데이터를 수집한다
* RSS(XML)를 파싱하여 내부 표준 형태(JSON)로 변환한다
* 변환된 결과를 API 응답으로 제공한다

연동은 HTTP 기반이며,
외부 응답 형식(RSS/XML)은 서버 내부에서만 처리한다.

---

## 4. 데이터 변환(표준 모델)

외부 RSS 항목을 내부 표준 모델로 변환한다.

내부 표준 모델(뉴스 항목) 최소 필드는 다음과 같다.

* id            : 식별자(서버에서 생성하거나 해시로 구성)
* title         : 제목
* link          : 원문 링크
* publisher     : 제공자(가능한 경우)
* publishedAt   : 발행 시각(가능한 경우)

내부 표준 모델은
공통 응답 포맷의 data 영역에 포함되어 반환된다.

---

## 5. API 제공 범위

뉴스 기능은 게시판 도메인과 분리된 별도 기능으로 제공한다.

* 뉴스 목록 조회 API 제공
* 검색/필터링은 선택 기능(기본 범위 제외)

---

## 6. 장애/실패 처리 정책

외부 시스템은 우리 시스템이 통제할 수 없으므로
다음 정책을 적용한다.

* 외부 호출 실패 시, 서버는 표준 오류 응답을 반환한다
* 실패는 전역 예외 처리 규칙에 따라 HTTP 상태코드로 표현한다
* 외부 응답 지연에 대비해 호출 시간 제한을 둔다

---

## 7. 캐시/저장 정책(선택)

외부 RSS는 요청마다 호출하지 않고
캐시/저장 정책을 적용할 수 있다.

* 기본: 일정 주기로 수집 후 저장
* 옵션: 마지막 성공 데이터를 캐시로 유지

캐시 저장소는 구현 단계에서 확정한다.

---

## 8. 책임 분리(역할)

* Controller: 뉴스 조회 API 제공(응답 변환)
* Service: 수집/변환 흐름 제어, 정책 적용
* Repository(선택): 저장/조회가 필요할 경우 데이터 접근

외부 호출은 Service의 책임으로 둔다.

---

## 9. 이 단계의 정리

타 시스템 연동 설계 단계에서는

* 연동 대상
* 연동 방식
* 내부 표준 모델
* 실패/캐시 정책
* 역할 분리

를 확정한다.

다음 단계에서는
이 설계를 바탕으로 **통합 구현(환경 구축 및 프로젝트 초기화)** 을 시작한다.
