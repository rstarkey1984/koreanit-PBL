# 01. 현행 시스템 분석

이 문서는 본 프로젝트를 시작하게 된 **실제 현행 시스템의 문제 상황**을 정리한다.

본 단계에서는 해결 방법이나 구현 기술을 다루지 않는다.
오직 **왜 구조 개선이 필요한지**를 명확히 하는 것이 목적이다.

---

## 1. 현행 시스템의 출발점

현행 시스템은 **Spring Boot 기반의 단일 컨트롤러 중심 구조**로 시작되었다.

* 하나의 Controller 클래스에 대부분의 API가 구현됨
* 빠른 구현을 위해 모든 로직을 한 파일에 집중
* 학습·개인 프로젝트에서 흔히 선택되는 방식

초기 단계에서는 구조보다 **동작 여부**가 우선되었고,
이 방식은 실제로 빠르게 결과를 만들어냈다.

---

## 2. 하나의 파일에 집중된 역할

단일 Controller 클래스는 다음과 같은 역할을 동시에 수행하고 있었다.

* HTTP 요청 매핑 및 응답 처리
* 요청 파라미터 검증
* 인증/세션 처리
* 비즈니스 로직 처리
* 데이터베이스 커넥션 획득
* SQL 작성 및 실행
* ResultSet 가공 및 응답 데이터 구성
* 공통 응답 포맷 처리

즉, 하나의 클래스가
**요청 처리부터 데이터 접근까지 모든 책임을 보유**하고 있었다.

---

## 3. 구조적 문제의 발생

기능이 추가되고 코드가 증가하면서 다음과 같은 문제가 나타났다.

### 3-1. 코드 가독성 저하

* 클래스 하나의 길이가 과도하게 증가
* 특정 기능의 시작과 끝을 파악하기 어려움

### 3-2. 수정 영향 범위 확대

* 작은 기능 수정에도 여러 메서드를 동시에 확인해야 함
* 로직 변경이 다른 API에 영향을 미칠 가능성 증가

### 3-3. 중복 코드 증가

* 인증, 예외 처리, 응답 포맷 관련 코드가 반복됨
* 공통 로직을 재사용하기 어려운 구조

---

## 4. 유지보수 관점의 한계

이 구조는 다음과 같은 한계를 가진다.

* 역할별 테스트가 어렵다
* 특정 로직만 단위로 검증하기 힘들다
* 기능 추가 시 기존 코드의 복잡도가 계속 증가한다

문제의 원인은
개별 개발자의 숙련도가 아니라,
**역할 분리에 대한 구조적 기준이 없었다는 점**이다.

---

## 5. 개선이 필요한 이유

본 프로젝트에서는 다음과 같은 방향의 개선이 필요하다.

* 각 책임을 명확히 분리할 수 있는 구조
* 요청 처리, 로직 처리, 데이터 접근의 역할 구분
* 기능 추가 시 기존 코드에 미치는 영향을 최소화

---
