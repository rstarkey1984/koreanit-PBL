# 1. 현행 시스템 분석 

> 현행 시스템 분석이란    
> “사람이 이 시스템에서 무엇을 하고, 그 결과로 무엇이 저장되는지 정리하는 것”이다.

## 1-1. 현행 시스템 분석을 하는 이유

- 지금 이미 존재하는 시스템이 어떻게 동작하는지 파악하는 단계

- 새로 만드는 프로젝트라도, 기준이 되는 서비스는 항상 존재함

예)
- 네이버 카페

- 학교 홈페이지 게시판

- 우리가 평소 쓰는 커뮤니티


## 1-2. 현행 시스템 분석에서 보는 것

### 1) 사용자가 무엇을 할 수 있는가? (기능 관점)
> 이 시스템에서 가능한 행동들 ( 버튼 기준으로 생각하면 이해가 빠름 )

- 회원가입이 있는가?

- 로그인은 어떻게 하는가?

- 글 작성은 누가 할 수 있는가?

- 글 삭제는 작성자만 가능한가?

### 2) 사용자가 어떻게 이동하는가? (화면 관점)
> 화면과 화면 사이의 이동 흐름

- 어떤 화면들이 존재하는가?

- 화면 이동 순서는 어떻게 되는가?

예)

- 목록 → 상세 → 목록

- 로그인 → 글쓰기 → 목록

> “사용자 동선”을 그려보는 단계

### 3) 사용자의 행동이 무엇으로 남는가? (데이터 관점)
> 사용자의 행동 중 DB에 저장되는 것

- 어떤 정보가 저장되는가?

- 글에는 어떤 컬럼이 필요할까?

- 조회수는 언제 증가할까?

> “이 행동이 DB에 어떤 변화 를 일으키는가?”


## 현행 시스템 분석 예시
```
게시판 시스템 분석

1. 시스템 개요

시스템 유형: 게시판 시스템
시스템 목적: 사용자 간 게시글과 댓글을 통한 정보 공유
기준 서비스: 학교 홈페이지 게시판, 일반 커뮤니티 게시판


2. 사용자 구분

비회원
- 글 목록 조회 가능
- 글 상세 조회 가능

회원
- 글 작성 가능
- 댓글 작성 가능

작성자
- 본인이 작성한 글 수정 가능
- 본인이 작성한 글 삭제 가능

관리자(존재할 수 있음)
- 모든 글 삭제 가능


3. 주요 화면 구성

글 목록 화면
- 게시글 전체 목록 표시

글 상세 화면
- 게시글 본문 및 댓글 표시

글 작성 화면
- 로그인 사용자만 접근 가능

글 수정 화면
- 작성자만 접근 가능

로그인 화면
- 사용자 인증 처리

회원가입 화면
- 신규 사용자 등록


4. 사용자 행동(기능)

- 회원가입
- 로그인 / 로그아웃
- 게시글 작성
- 게시글 수정
- 게시글 삭제
- 댓글 작성
- 댓글 삭제
- 게시글 목록 조회
- 게시글 상세 조회


5. 화면 이동 흐름

글 읽기 흐름
글 목록 → 글 상세 → 글 목록

글 작성 흐름
글 목록 → 로그인 → 글 작성 → 글 상세(또는 목록)

댓글 작성 흐름
글 목록 → 글 상세 → 댓글 작성 → 글 상세


6. 데이터 저장 구조

회원 정보
- 아이디
- 비밀번호(해시값)
- 이름 또는 닉네임
- 가입일

게시글 정보
- 작성자 ID
- 제목
- 본문 내용
- 조회수
- 작성일

댓글 정보
- 게시글 ID
- 작성자 ID
- 댓글 내용
- 작성일


7. 데이터 관계

회원 1명은 여러 개의 게시글을 작성할 수 있다
게시글 1개에는 여러 개의 댓글이 달릴 수 있다


8. 주요 정책(규칙)

비회원은 게시글 작성이 불가능하다
게시글 수정 및 삭제는 작성자만 가능하다
댓글 삭제는 작성자만 가능하다
게시글 조회수는 상세 페이지 접근 시 증가한다


9. 요약 정리

사용자는 화면에서 행동을 하고
그 행동에 따라 화면이 이동하며
그 결과는 데이터베이스에 저장된다
```

# 2. 요구사항 확인

## 2-1. 요구사항이란?

- 이 시스템이 반드시 해야 하는 일

- 프로젝트 진행 중 변경 시 일정과 범위에 큰 영향을 주는 사항

## 2-2. 요구사항의 종류

### 1) 사용자 요구사항
> 사용자 요구사항: 사용자가 직접 하는 일    
- 회원은 글을 작성할 수 있어야 한다

- 로그인하지 않으면 글을 작성할 수 없다

- 댓글은 게시글에 종속된다

### 2) 시스템 요구사항
> 시스템 요구사항: 시스템이 내부적으로 지켜야 할 규칙
- 로그인 상태는 세션으로 관리한다

- 데이터는 DB에 저장한다

- 삭제 시 관련 데이터도 함께 삭제된다

## 2-3. 요구사항 작성 시 주의점
> 조건과 제약이 반드시 들어가야 함
잘못된 예

- “게시판을 만든다”

- “로그인을 구현한다”

올바른 예

- “로그인한 사용자만 게시글을 작성할 수 있다”

- “본인이 작성한 글만 삭제할 수 있다”



## 요구사항 명세서 작성 예시:

```
요구사항 명세서

1. 프로젝트 개요

프로젝트명: 게시판 웹/API 서비스

목적:
회원 인증을 기반으로 게시글과 댓글을 관리하는 게시판 시스템을 구현한다.

시스템 설명:
비회원은 게시글을 조회할 수 있으며,
로그인한 사용자는 게시글과 댓글을 작성 및 삭제할 수 있다.
모든 기능은 웹 또는 API 형태로 제공된다.


2. 사용자 요구사항

1) 사용자는 회원가입을 할 수 있어야 한다.
2) 사용자는 로그인 및 로그아웃을 할 수 있어야 한다.
3) 비회원은 게시글 목록 및 게시글 상세를 조회할 수 있어야 한다.
4) 로그인한 사용자만 게시글을 작성할 수 있어야 한다.
5) 게시글은 최신순으로 목록 조회되어야 한다.
6) 게시글 상세 조회 시 조회수가 증가해야 한다.
7) 게시글은 작성자 본인만 수정할 수 있어야 한다.
8) 게시글은 작성자 본인만 삭제할 수 있어야 한다.
9) 댓글은 로그인한 사용자만 작성할 수 있어야 한다.
10) 댓글은 특정 게시글에 종속되어야 한다.
11) 댓글은 작성자 본인만 삭제할 수 있어야 한다.


3. 시스템 요구사항

1) 로그인 상태는 세션 기반으로 관리되어야 한다.
2) 게시글 삭제 시 해당 게시글에 종속된 모든 댓글은 함께 삭제되어야 한다.
3) 게시글 삭제와 댓글 삭제는 하나의 트랜잭션으로 처리되어야 한다.
4) 모든 데이터는 MySQL 데이터베이스에 저장되어야 한다.
5) 모든 API 응답은 JSON 형식이어야 한다.
6) 요청 파라미터가 누락되거나 형식이 올바르지 않은 경우 오류 응답을 반환해야 한다.
7) 로그인하지 않은 사용자가 인증이 필요한 요청을 할 경우 오류 응답을 반환해야 한다.
8) 작성자가 아닌 사용자가 게시글 또는 댓글 삭제/수정 요청을 할 경우 오류 응답을 반환해야 한다.


4. 기타 정책

1) 게시글 조회수는 게시글 상세 페이지 요청 시 1씩 증가한다.
2) 비정상적인 요청에 대해서는 공통 오류 형식을 사용한다.


5. 요약

사용자는 시스템을 통해 게시글과 댓글을 조회하고,
로그인 상태에 따라 작성 및 삭제 권한을 가진다.
사용자의 모든 행동은 데이터베이스에 기록되며,
시스템은 권한과 데이터 무결성을 보장해야 한다.
```

# 3. 분석 모델 

## 분석 모델이란?

- 요구사항을 그림과 구조로 바꾼 것

- 개발자 혼자만 이해하는 문서가 아니라,

  - 팀원

  - 강사

  - 평가자  
    모두가 이해할 수 있어야 함


## 3-1. 기능 관점 분석모델

### 기능 목록 / 기능 분해

- 시스템이 제공해야 할 기능을 목록으로 정리

- 요구사항 누락 여부를 확인하는 데 사용

예

- 회원: 회원가입, 로그인

- 게시글: 목록, 상세, 작성, 삭제

- 댓글: 작성, 목록, 삭제

> 기능 관점 분석모델은  
> “무엇을 만들어야 하는가”를 확인하는 도구다.

### 기능 관점 분석모델 예시:
```
[회원]
- 회원가입
- 로그인
- 로그아웃

[게시글]
- 목록 조회
- 상세 조회
- 작성
- 삭제

[댓글]
- 작성
- 목록 조회
- 삭제
```

## 3-2. 사용자 시나리오 관점 분석 모델
> 사용자 시나리오는 “사용자가 시스템을 실제로 쓰는 모습”을 글로 정리한 분석 모델이다.   
> 즉, 요구사항을 실제 사용 흐름으로 풀어 쓴 형태


사용자 시나리오에서 확인하는 것

- 사용 순서가 논리적인가?

- 로그인 여부에 따른 분기가 드러나는가?

- 사용자 요구사항을 모두 포함하고 있는가?

- 불필요한 기능이나 누락된 기능은 없는가?

사용자 시나리오 작성 규칙

- 번호를 붙여 순서대로 작성한다

- 한 단계에 하나의 행동만 작성한다

- 조건(로그인 여부 등)을 문장에 명확히 포함한다

- 구현 방법은 작성하지 않는다

### 사용자 시나리오 분석 모델 예시:
```
시나리오 A: 게시글 작성 흐름

1. 사용자는 게시글 목록을 조회한다.
2. 글쓰기 버튼을 선택한다.
3. 로그인하지 않은 경우 로그인 화면으로 이동한다.
4. 로그인을 완료하면 게시글 작성 화면으로 이동한다.
5. 게시글을 작성한다.
6. 게시글 목록 화면으로 이동한다.

시나리오 B: 게시글 조회 + 댓글 작성 흐름

1. 사용자는 게시글 목록을 조회한다.
2. 게시글을 선택하여 상세 페이지로 이동한다.
3. 로그인하지 않은 경우 로그인 화면으로 이동한다.
4. 로그인을 완료하면 게시글 상세 페이지로 이동한다.
5. 댓글을 작성한다.
6. 댓글이 포함된 게시글 상세 페이지를 확인한다.
```

## 3-3. 데이터 관점 분석모델

ERD (초안)

- 어떤 테이블이 필요한지

- 테이블 간 관계는 어떻게 되는지 표현

예

- users

- posts

- comments

관계

- 사용자 1 : 게시글 N

- 게시글 1 : 댓글 N

> 이 단계에서는 컬럼 타입보다 관계 이해가 더 중요하다.

# 🧩 실습 / 과제

## 실습 과제 목표

- 요구사항을 문서로 정리한다

- 요구사항을 분석 모델로 구조화한다

- “생각 → 구조 → 설계 준비” 흐름을 경험한다

## 분석 과제로 쓰기 좋은 실제 서비스 예시

- 네이버 블로그

- 네이버 카페

- 배달의민족 (리뷰 기능 한정)

## 1. 요구사항 명세서 작성

### 작성 내용

- 프로젝트 개요

- 사용자 요구사항

- 시스템 요구사항

### 작성 기준

- A4 1장 이내

- 문장 형태로 작성

- 구현 방법(API, SQL 등)은 작성하지 않는다

### 평가 포인트

- 사용자 요구사항과 시스템 요구사항이 구분되어 있는가

- 조건과 제약이 명확한가

- 모호한 표현이 없는가


## 2. 분석 모델 작성

### 1) 기능 관점 분석모델

- 기능 목록 또는 기능 분해 형태

- 회원 / 게시글 / 댓글 단위로 정리

### 2) 사용자 시나리오 관점 분석모델

- 대표 사용자 시나리오 2개 이상

- 로그인 여부에 따른 흐름 포함

- 번호를 붙여 순서대로 작성

### 3) 데이터 관점 분석모델

- ERD 초안

- 테이블 이름과 관계(1:N) 표현

- 컬럼 타입은 생략 가능
